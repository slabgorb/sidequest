# Story 14-7: Image scene relevance filter — validate art prompt matches current narrative context

## Story Details
- **ID:** 14-7
- **Epic:** 14 (Multiplayer Session UX)
- **Workflow:** tdd
- **Priority:** p1

## Story Summary

Before sending an art prompt to the daemon, validate that the subject matter matches the current scene context (location, NPCs present, action type). Reject prompts that reference entities not in the scene. Use the scene interpreter's subject list as the ground truth for what should appear in the image.

## Success Criteria

- [ ] Scene relevance validator extracts entity list from scene interpreter
- [ ] Image prompt validation checks subject entities against scene context
- [ ] Invalid prompts are rejected before reaching daemon
- [ ] Valid prompts proceed normally to image generation
- [ ] Tests cover positive (valid prompts) and negative (invalid prompts) cases
- [ ] DM can force generation to bypass validation (emergency override)

## Technical Context

### Image Generation Pipeline (Current Flow)

1. **dispatch_player_action()** in `sidequest-server/src/lib.rs` (lines ~4432-4498):
   - Narration text generated by Claude CLI subprocess
   - `SubjectExtractor::extract()` parses narration → RenderSubject with entities list
   - `BeatFilter` evaluates if render should proceed (narrative weight, cooldown, burst)
   - **If approved:** `queue.enqueue(subject, art_style, model, neg_prompt)`
   - Render job queued to background worker

2. **RenderQueue** in `sidequest-game/src/render_queue.rs`:
   - Accepts RenderSubject with: entities (Vec<String>), scene_type, tier, prompt_fragment
   - Content hash dedup based on entities + scene_type + tier
   - Spawns render worker that:
     - Calls daemon HTTP API to generate image
     - Returns RenderJobResult (Success/Failed) via broadcast channel

3. **Render Integration** in `sidequest-server/src/render_integration.rs`:
   - `spawn_image_broadcaster()` listens for RenderJobResult
   - Converts to IMAGE message payload
   - Broadcasts to all WebSocket clients

### RenderSubject Interface

```rust
pub struct RenderSubject {
    entities: Vec<String>,      // ["Orc", "Goblin", "Tavern Sign"]
    scene_type: SceneType,      // Combat | Dialogue | Exploration | Discovery | Transition
    tier: SubjectTier,          // Portrait | Scene | Landscape | Abstract
    prompt_fragment: String,    // "An orc and goblin in a tavern"
    narrative_weight: f32,      // 0.0-1.0 (significance)
}
```

Getter methods: `entities()`, `scene_type()`, `tier()`, `prompt_fragment()`, `narrative_weight()`

### ExtractionContext (Available to Relevance Filter)

```rust
pub struct ExtractionContext {
    pub known_npcs: Vec<String>,        // From NPC registry
    pub current_location: String,       // Extracted from narration header
    pub in_combat: bool,                // From CombatState
    pub recent_subjects: Vec<String>,   // (for dedup, unused)
}
```

### What Constitutes "Scene Context"

From the game engine's perspective (sidequest-game):
- **Location** — extracted from narration "header" (first sentence pattern match)
- **Active NPCs** — from `npc_registry` (NPC names mentioned in recent narration)
- **Entities from RenderSubject** — the extracted list of who/what should be in the image
- **Scene Type** — Combat | Dialogue | Exploration | Discovery | Transition

### Validation Logic Required (14-7)

**Before** queuing to render daemon, validate:

1. **Location Match** — Prompt mentions should be in the current location OR location should be transitioning
2. **NPC Presence** — Any character/NPC mentioned in prompt must be in `known_npcs` OR be a generic (goblin, orc, dragon)
3. **Scene Type Alignment** — SceneType should match the prompt (combat prompts during exploration = suspicious)
4. **Action Consistency** — If narration says "hiding" but prompt says "standing in the middle of battle" = invalid

**Allow override:** If DM explicitly requests image via `/render force`, bypass validation

### Story 14-6 Integration

Story 14-6 (Image pacing throttle) introduces `image_cooldown_seconds` config and suppresses rapid-fire image triggers.

This story (14-7) runs **after** pacing filter succeeds. Both should work in sequence:
1. **Cooldown filter** (14-6) — Is it too soon since the last image?
2. **Relevance filter** (14-7) — Does the image match the scene?
3. **Render enqueue** — If both pass, queue the job

### Known Challenges

- **Generic vs Specific Entities** — "goblin" vs "Grok the Goblin" — should both be valid if Grok is in known_npcs
- **Narrative Vagueness** — Narration might say "the stranger" or "the hooded figure" without naming NPCs; should this be rejected or allowed?
- **Location Extraction** — Current logic uses regex on narration header; may miss multi-location scenes
- **Scene Type Drift** — LLM might generate dialogue prompts during combat (happens with narrator drift)
