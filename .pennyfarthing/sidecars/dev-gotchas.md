### Git discipline
- **`cd` persists within a single Bash call.** `cd foo && git -C foo …` resolves to `foo/foo`; the shell resets only *between* calls. Use absolute paths with `git -C` throughout (never mixing a `cd`), or re-anchor with `cd <subdir>` at the start of each chain.
- **Hard ban on `git stash`** — no exceptions, including read-only diagnostics and post-merge "safe" cases. Stashes poof; branches are durable. Set work aside via `git switch -c tmp-<reason>` + commit; to peek upstream use `git show origin/<branch>:<path>`.

### Postgres substrate (ADR-115) traps
- **Secondary writes inside an open turn transaction MUST ride that tx, or they self-deadlock under PG (2026-05-26, D8).** `emit_event` opens `with repo.transaction() as tx:` which takes a per-session `FOR UPDATE` row lock on `sessions` (connection A). Any telemetry/side-write reached *inside* that block (e.g. the projection fan-out's visibility-gated `_publish_secret_routed` → `watcher_hub.publish_event` → `PgTelemetrySink.record`) that opens its OWN pooled connection takes a SECOND `session_tx` `FOR UPDATE` on the same locked row → self-deadlock. **Harmless on SQLite** (one in-process connection, no row lock) so it only surfaces under PG on a real MP turn. Confirm your own changed *source* (non-test) files add zero `tsc` errors: `npm run build 2>&1 | grep 'error TS' | grep -v '__tests__\|\.test\.'`. The enforced gate here is **vitest + lint**, not `tsc -b`.

### Routing a file through a strict typed loader surfaces previously-tolerated invalid fixtures (2026-06-03, story 65-12)
- **What happened:** The lore reference page rendered `legends.yaml` via the generic humanization walk (`_file_renders_by_stem`), which never validates — it just humanizes raw YAML. 65-12's Timeline reads legends through the **typed** loader (`genre/loader._load_legends_flexible` → `Legend.model_validate`, `extra="forbid"`). Calling it unconditionally in `assemble_lore_page` made **19 pre-existing reference tests** 500, because their seed worlds authored `Legend`-invalid legends (`- name: x\n  origin: unknown` — `origin` is a forbidden field).
- **Why it matters:** The instinct is "my feature broke tests, weaken the loader to tolerate it." That's wrong here. `_load_legends_flexible` is the SAME loader the genre loader runs at real world-load (`loader.py:1048`, **no try/except — fatal**). So a legends.yaml the typed loader rejects describes a world that **cannot be played at all**; the reference page 500ing on it is correct (consistent with world-load), and the seed fixtures were authoring engine-impossible content that only "worked" because the old reference path never validated. Weakening the loader would also have broken the story's own No-Silent-Fallbacks AC.
- **How to apply:** When you newly route an existing file through a stricter/typed loader, expect to surface fixtures (and possibly real content) that the old lenient path tolerated. Before "fixing" by loosening: check whether the **production** load path already validates strictly (grep the genre loader / `model_validate` call sites). If production is already strict, the fixtures are the bug — correct them to valid content. If production is lenient there too, then a real world could hit it and you must decide loud-fail vs tolerate per the story's No-Silent-Fallbacks contract. **Verify pre-existing-ness honestly:** `git stash -u` your implementation and re-run a sample of the "new" failures — if they still fail, they're not yours (here, 27 full-suite failures were all pre-existing Epic 74-3 content blockers outside `tests/server/`).
- **Chrome-wiring coupling:** the reference page has a keystone test (`test_reference_chrome_wiring.py`) asserting every emitted `class="..."` token has a selector in the served CSS bundle (`static/reference/{theme,styles,presenters}.css`). A new rendered section MUST ship CSS rules for all its classes AND have a chrome-wiring seed world that actually emits the full class set, or the keystone fails. Add the CSS in the same phase you add the section.

### Long multi-phase sessions risk concurrent-landing rebase conflicts at finish; `merge_pr` can over-report on a draft/conflicting PR (2026-06-03, story 65-12)
- **What happened:** Over a long single session, 65-12 ran RED→GREEN→spec-check→verify→review→reconcile→finish. While it ran, 65-13 (#614) and 64-16 merged into `develop`, touching the same files (`telemetry/spans/reference.py`, `reference_renderer.py`, `test_reference_chrome_wiring.py`). `pf sprint story finish` reported `merge_pr` done and marked the story `done` + archived the session, but PR #615 was **draft + CONFLICTING** and never actually merged — the code was NOT on develop.
- **Why it matters:** A *reported* finish is not a *merged* finish. Trust `gh pr view <n> --json state,mergedAt,mergeable,isDraft` over the finish script's step list. A draft PR can't auto-merge; concurrent landings on shared files conflict. The dangerous desync is sprint-YAML=`done` while `develop` lacks the code — hold the sprint commit until the merge truly lands.
- **How to apply (rebase-recovery):** `gh pr ready <n>` → `git fetch origin develop && git rebase origin/develop`. For span/registry files where two stories each *added* entries the conflict is "keep both" (both constant groups, both `FLAT_ONLY_SPANS` entries, both constructor fns; watch the case where the diff splits a function mid-`Span.open` — give each fn its own complete `tracer_override=... ) as span: yield span` tail). `ast.parse` + grep for markers, re-run the suite (now includes the other story's tests), `git push --force-with-lease`, `gh pr merge --squash --delete-branch`, verify `mergedAt` set + file exists on `origin/develop`.
- **Hook note:** with the server repo checked out on `develop`, the `pf hooks` PreToolUse guard blocks Bash that looks like it could commit to a protected branch (even a chained `cat >> ... && pf ...`). Run `pf` commands standalone; do file edits via the Edit/Write tools.

### Adding a new genre_pack attribute read in a shared consumer breaks MagicMock-pack fixtures (2026-06-03, story 82-8)
- **What happened:** 82-8 added a `sd.genre_pack.progression.wealth_tiers` read inside `views.party_member_from_character` (a consumer many tests drive). My new file's tests used a *real* loaded pack and passed, but two tests in `tests/server/test_reference_url_attach.py` build `genre_pack = MagicMock()` and pin only the attributes the consumer reads. They had no `progression.wealth_tiers`, so it auto-mocked to a MagicMock. `resolve_wealth_tier(gold, <MagicMock>)` then misfired: `not MagicMock()` is False (skips the empty-guard), `for x in MagicMock()` iterates empty (MagicMock.__iter__ defaults to `iter([])`), so the loop fell through to `return tiers[-1]` → `MagicMock.__getitem__` returns a truthy MagicMock (not None), and back in views `next(i for i,t in enumerate(tiers) if t is resolved_tier)` over the empty mock raised **StopIteration**.
- **Why it matters:** A MagicMock genre_pack silently satisfies *any* attribute access, so a new consumer read doesn't fail loudly at the mock — it produces garbage that detonates deeper in. The full-suite run is the only thing that catches it; a focused run of just your new file is green and lies. ALWAYS run the full server suite (`uv run pytest -q`) after touching a shared view/consumer, not just your story's file.
- **How to apply:** When you add a `genre_pack.<x>` read to a shared consumer, grep the test tree for synthetic `genre_pack = MagicMock()` fixtures (`grep -rn "genre_pack = MagicMock" tests/`) and pin the new attribute to a realistic empty value (`genre_pack.progression.wealth_tiers = []`). This is the established precedent in that exact file — story 68-1 added `genre_pack.rules.survivability_pool_label = None` for the same reason. It's a fixture update, NOT a test weakening (the affected tests' own assertions stay untouched), and NOT a license to add production defensiveness — real packs always carry a real `ProgressionConfig` with list defaults, so the production path can't hit the mock pathology. Keep `resolve_*` helpers returning an element of the input list so an `is`-identity `next()` over that list can't StopIteration in any real path.
