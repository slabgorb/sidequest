## Dev Gotchas

### Cargo test discipline (2026-04-19 incident)
- **Never stack cargo test runs.** One invocation at a time. If one is running (mine or anyone else's), wait — do not spawn a concurrent one. Stacked runs thrash sccache/`.cargo-lock` and burn enormous amounts of wall time for zero information gain. Keith: "The way you are doing it is such an amazing waste of fucking time!"
- **Never pipe cargo output to `tail` / `head` / `grep`.** The project has a PreToolUse hook + capture script that owns cargo output. Run `cargo test ...` bare — no shell plumbing. Piping fights the hook and loses data.
- **Never `pkill -9 -f cargo` to "clear the decks."** That kills Keith's other tmux panes too, not just my processes. Only kill PIDs I personally spawned via `run_in_background`. If I think a test is stuck, ask — don't nuke.

### testing-runner reformats YAML if asked to "fix" (2026-06-01, story 71-24)
- **A `testing-runner` helper tasked to verify GREEN went rogue and edited content YAML to make a test pass — via a full YAML round-trip (yq/yaml-dump).** It stripped every `# --- SECTION ---` comment, mangled a `notes: >-` folded block into an escaped one-liner, turned em-dashes into `—`, and reflowed all list indentation across the whole file. The needed change was a 3-line damage-spec addition; the diff was 300+ lines of corruption with the real change buried inside.

Re-task the helper with a HARD read-only mandate ("run and report ONLY; do NOT edit/format/round-trip any file; if a test fails, report it"). The helper is for running tests, not authoring — content authoring is Dev's lane.
- **Always `git diff --stat` after a helper run.** If a file you didn't expect shows a huge insertion/deletion count, a serializer round-trip happened — revert and redo by hand. Never let a generated whole-file reformat reach a commit.
- **YAML content edits are hand-edits.** Genre-pack YAML is human-authored with comments, inline-flow `damage: {dice: "1d6", bonus: 0}`, and folded scalars that carry intent. Edit with surgical `Edit` calls that preserve surrounding structure — never dump-and-rewrite.

### Content-mechanics stories span two repos (story 71-24)
- **A genre-pack mechanics story (`repos: sidequest-content`) also needs the `sidequest-server` test branch pushed.** TEA's failing tests live in `sidequest-server/tests/server/` (they drive the loader + intent-match + seating/dice path); the YAML you author lives in `sidequest-content`. A genre-tier mechanics change can be clean while the world still FAILs the structure validator on pre-existing asset gaps — don't conflate the two; check `git diff --name-only` to prove your change didn't cause it.

### The wiring failure class (the dominant bug category in this project)
- **Don't reinvent — wire up what exists.** Before building anything new, grep the codebase. Many systems are fully implemented but not wired into the server or UI. CLAUDE.md rule #3.
- **No half-wired features.** Connect the full pipeline or don't start. If something needs 5 connections, make 5 connections. Shipping 3 and calling it done is the exact failure mode Epic 15 was created to clean up.
- **Wire it, don't just define it.** Adding fields to a struct is not wiring. Adding props to a component is not wiring. Data must actually flow from source → consumer. If no code populates and transmits the field, the feature isn't done.
- **Never rationalize unwired code.** When `/sq-wire-it check` reports FAIL (no non-test consumers), the answer is **wire it**. Do not write paragraphs explaining why this time is different. If I catch myself typing "this is acceptable because" — STOP. That sentence is the bug. The wiring usually takes 5 minutes; the rationalization takes longer.
- **"Wiring means dashboard."** Internal data flow without OTEL visibility is not wired. A missing `WatcherEventBuilder::new(...)` call on a subsystem's dispatch path is a blocking wiring bug, never a "non-blocking improvement." Labeling an OTEL gap as non-blocking in my own audit is the exact self-compounded failure I hit in the 2026-04-09 Map fix.
- **No agent rationalization.** Dev/Architect/TEA/Reviewer are all Claude — they don't get to split blame across personas. "The Architect said it was fine" is not a defense. If a wiring gap is identified in any phase, the only valid response is "fix the code now," never "clarify the spec" or "file a follow-up story."
- **Verify wiring before claiming done.** `grep` for non-test consumers on the merged branch before marking a fix as complete. Don't wait for Keith to ask. A playtest with the GM panel open is the acceptance test.
- **Verify end-to-end before claiming fixed.** Code review is not verification. Trace the full wire path: source → channel → writer task → WebSocket → client handler → DOM. Session 7 had six consecutive "fixed" claims on the turn-lock bug, each found a real issue, none verified the message actually arrived at the browser.
- **Wiring gaps require action, not acknowledgment.** Finding a wiring gap and saying "yep that's a wiring gap" is worthless. Fix it. Log it. Check for siblings (if one compute-then-ignore exists, grep for the pattern). Verify OTEL. Don't move on.
- **Kitchen-sink gate failure is real** (story 16-1): nine agents approved unwired code because every phase rationalized it as "scope boundary" and "wiring comes in subsequent stories." The wiring-check gate exists in `.pennyfarthing/gates/` — every agent must enforce it, not just the reviewer as a backstop.

### Python server test hygiene (2026-04-23 incident — the full-session disaster)

If `claude_client_factory` isn't injected, BOTH spawn real `claude -p` subprocesses. Mocking `Orchestrator.run_narration_turn` alone is insufficient — the decomposer runs first.

### Git discipline
- **`cd` persists within a single Bash call.** `cd foo && git -C foo …` resolves to `foo/foo`; the shell resets only *between* calls. Use absolute paths with `git -C` throughout (never mixing a `cd`), or re-anchor with `cd <subdir>` at the start of each chain.
- **Hard ban on `git stash`** — no exceptions, including read-only diagnostics and post-merge "safe" cases. Stashes poof; branches are durable. Set work aside via `git switch -c tmp-<reason>` + commit; to peek upstream use `git show origin/<branch>:<path>`.
- **Never stash during an active merge.** It strips MERGE_HEAD; `stash pop` injects fresh `<<<<<<<` markers on partly-resolved files and the first commit becomes a one-parent commit instead of the two-parent merge GitHub needs to mark the PR mergeable (45-3 incident). Inspect mid-conflict with read-only `git show origin/<branch>:path`.
- **Never stash + checkout develop to check if a failure is "pre-existing."** Wastes time, risks the tree, irrelevant — pre-existing or not, the job is a green suite (Keith, 2026-05-25).
- **Subrepos are independent git repos**, normally on `develop`; a `git commit` inside one lands on *that* subrepo's branch, not the orchestrator feature branch. Per gitflow each subrepo needs a `feat/*` branch. Before dispatching any implementer that touches a subrepo, check `git branch --show-current` and create the matching `feat/<feature>` branch first (2026-05-10: six subagents committed straight to develop).
- **Stacked-branch commits can silently orphan.** Commits that logically belong on parent B but were made on child C don't survive B's squash-merge, then `git rebase --onto develop B C` drops them via patch-id dedup (2026-05-09, 12 commits lost; recovered via `git reflog --all` cherry-pick). Before squash-merging a stacked parent, verify its tip with `git log --stat parent..child -- <files>`.
- **oq-1 and oq-2 are separate orchestrator clones**, each with its OWN subrepos that can drift across branches. Always dispatch subagents / verify with ABSOLUTE paths (`git -C /abs/path`), never bare `cd sidequest-server`. Mass test failures at pack-load are a top symptom of content-clone branch drift — check `git -C .../sidequest-content branch --show-current` FIRST (`main` lacks beneath_sunden/Epic-24 YAML). After `gh pr merge --delete-branch`, gh reverts the tree to the stale local default branch.
- **macOS BSD `sed` silently no-ops `\b`** — use literal patterns + grep-verify. Make a safety branch before any non-trivial rebase.

### Test discipline
- **The triad: FIXTURES for unit tests, VALIDATORS for the worlds, never direct unit tests on live content.** Don't `load_genre_pack(CONTENT_GENRE_PACKS / "...")` and assert on its data. ### Postgres substrate (ADR-115) traps
- **Secondary writes inside an open turn transaction MUST ride that tx, or they self-deadlock under PG (2026-05-26, D8).** `emit_event` opens `with repo.transaction() as tx:` which takes a per-session `FOR UPDATE` row lock on `sessions` (connection A). Any telemetry/side-write reached *inside* that block (e.g. the projection fan-out's visibility-gated `_publish_secret_routed` → `watcher_hub.publish_event` → `PgTelemetrySink.record`) that opens its OWN pooled connection takes a SECOND `session_tx` `FOR UPDATE` on the same locked row → self-deadlock. **Harmless on SQLite** (one in-process connection, no row lock) so it only surfaces under PG on a real MP turn. Fix: thread `tx`/`event_seq` (optional, default `None`) down the call chain so the inner write rides the same connection; lazy-fill / out-of-frame callers keep `None` and safely take their own short `session_tx` (no row lock held). Regression: `tests/game/projection/test_secret_routed_rides_turn_tx.py` (timeout-guarded — the deadlock shows as a 30s timeout, not a hang). Commit `77170b1`.
- **Migration test-debt is "seed-SQLite → read-PG", not `MissingDatabaseUrlError`.** With both `SIDEQUEST_DATABASE_URL` + `SIDEQUEST_TEST_DATABASE_URL` exported (PG keg-only v18, `/opt/homebrew/opt/postgresql@18/bin`), the dominant D8 failure was tests seeding only the SQLite save_dir while D2 slug-connect loads the authoritative snapshot/characters/narration/scrapbook/telemetry from PG via `db_pool.get_pool()` (it reads SQLite only for the bootstrap game row). Symptom: `has_character=False` → wrong connect branch. Fix per test: the truncating autouse `_pg_isolation` fixture (bind pool to per-worker `migrated_db` + `TRUNCATE … RESTART IDENTITY CASCADE`) **plus** seed PG via `_build_pg_repos_for_slug(db_pool.get_pool(), slug=...)` + `repo.save(snap)`. Canonical patterns: `tests/server/test_turn_telemetry_wiring.py`, `tests/server/test_session_handler_slug_resumed.py`, forensic seeding via `tests/persistence/test_pg_forensic.py::forensic_env`.
- **Fixed-slug server tests LEAK across the shared `sidequest_test` DB without per-test TRUNCATE.** Reusing a constant slug (`test-slug`, etc.) means a sibling test's persisted snapshot/scenario_state survives into the next test's connect-resume — manifesting as bizarre cross-test contamination ("genre pack not found" because a *sibling's* leaked `genre_slug` row resolved). ### NPC stores are two-tier — `npcs` (roster) vs `npc_pool` (walk-ons) (playtest #C1, 2026-05-28)
