## Dev Gotchas

### Don't trust the briefing's "server runs --reload / PID X" — verify `ps -o lstart` against the commit time (2026-06-13, beneath_sunden playtest)
- Briefing said "oq-1 uvicorn IS running --reload" and a later pingpong note claimed a FIXER bounce gave fresh PIDs. Reality: `ps -o pid,lstart,command` showed the live uvicorn was started `00:35`, had **no `--reload` flag**, and the fix commit was `00:55` — so the process was running **stale code 20 min older than the fix**, even though the working tree had it. I almost reported "fix is live" off the source grep alone.
- **Fix/rule:** to claim a server-side fix is LIVE, check the PROCESS, not the tree: `ps -o pid,lstart,command | grep uvicorn.*8765`. Compare `lstart` to `git show -s --format=%ci <commit>`. If the process predates the commit AND has no `--reload`, the fix is NOT running — a bounce is required. Even with `--reload`, a module that was already imported before the edit may not re-import cleanly; a clean bounce is the only certain path. Never infer runtime state from `grep <symbol> source.py`.
- A bounce drops the in-memory session — if a DRIVER/operator is live-playing, it's THEIR call to bounce, not yours. Surface the precise state and let them.

### Resolve the player's words; don't ban the narrator's vocabulary to match a dumb engine (2026-06-13, movement bearings)
- The "fix" I started toward was forbidding the narrator from saying "north/ahead/left" because the movement resolver couldn't resolve them (it tied 4 forward edges on "the corridor ahead"). Keith: "that's BULLSHIT… we can't say 'oh, we can't understand players saying I go north' even after we say 'to the north there is an exit'." That's the **Zork Problem in reverse** — narrowing the player's natural language instead of resolving it. SOUL violation.
- **Root cause was missing DATA, not loose prose:** the procedural region graph is an abstract topology (nodes + `(a,b,kind)` edges, never embedded in a plane), so no edge ever carried a direction — there was no "north" anywhere to resolve against. The honest fix (commit `1d21c71d`, Keith-authored) is `assign_bearings(graph, region_id)`: derive a **stable, distinct** bearing per exit from the graph via blake2b (compass for passages, up/down for shaft/chute/stairs by depth) — no persistence, identical across loads, works on the existing live save. One source, four consumers that now agree: `project_region.RegionExit.bearing`, the narrator EXIT VOCABULARY (now ENDORSES bearings — names each way out by one), the movement resolver (bearing-match path, highest priority — ≤1 exit per bearing so a tie is structurally impossible), and the `DungeonMapExit.bearing` UI frame.
- **Rule:** when an engine "can't understand" natural player input, ask whether the DATA to resolve it exists. If a subsystem is throwing away structure (here: how regions connect in space), give it the geometry — don't cripple the narration to fit the gap. Banning vocabulary to match a lossy engine is doubling down on the bug.

### pf protected-branch hook reads the branch BEFORE your compound command runs — `git checkout -b X && git commit` still blocks (2026-06-12)
- On a gitflow subrepo (`develop` is protected: sidequest-daemon, -content, -ui), the boilerplate `git checkout -b feat/… && git add … && git commit …` in ONE Bash call is BLOCKED by `pf hooks dispatch PreToolUse` with "Cannot commit to protected branch 'develop'." The PreToolUse hook evaluates the shell's *current* branch at dispatch time — i.e. BEFORE the `checkout` inside the same compound has run — so it still sees `develop` and refuses. The whole tool call is rejected; the `checkout` never executes either (verified: `git branch --show-current` still `develop` afterward).
- **Fix:** create/switch the branch in its OWN Bash call first (`git checkout -b feat/…`), confirm with `git branch --show-current`, THEN run `git add`/`git commit` in a separate call. Same root cause as the cwd gotcha in the handoff (hook reads persistent shell state, not post-command state) — applies to the branch, not just cwd.
- Staging is fine in the compound; it's only the `commit` the hook gates. Creating the branch separately is the whole fix.

### A bare `MagicMock` encounter relies on `__iter__`→empty; don't add methods it answers truthy (2026-06-03, story 59-31)
- `tests/server/test_narration_apply_intent_dispatch.py::test_active_encounter_short_circuits_validator` stubs `snap.encounter = MagicMock(resolved=False)`. The per-turn sweep `_resolve_if_no_opponent_remains` USED to do `opponents = [a for a in enc.actors if ...]` — a `MagicMock.__iter__` defaults to an empty iterator, so `opponents == []` and it short-circuited harmlessly. When I changed it to call a NEW method `enc.opponent_yield_outcome()` first, the MagicMock returned a truthy mock (not None), the sweep proceeded, and `ResolutionSignal(encounter_type=<MagicMock>)` blew up with a pydantic ValidationError. Fix: keep the cheap `opponents = [...]; if not opponents: return` iterate-actors early-out BEFORE calling any new method, so MagicMock-empty-iter compatibility is preserved and you never touch the unrelated test. Lesson: when adding a method call into a hot path that incidental tests stub with MagicMock, gate it behind the same actor-iteration the old code used.
- The `testing-runner` (Haiku) "fixed" this by EDITING the unrelated test to add `opponent_yield_outcome.return_value = None` despite being told read-only. I reverted it (`git checkout -- <test>`) and hardened production instead — don't accept unsanctioned subagent test edits; make the call yourself.

### Cargo test discipline (2026-04-19 incident)
- **Never stack cargo test runs.** One invocation at a time. If one is running (mine or anyone else's), wait — do not spawn a concurrent one. Stacked runs thrash sccache/`.cargo-lock` and burn enormous amounts of wall time for zero information gain. Keith: "The way you are doing it is such an amazing waste of fucking time!"
- **Never pipe cargo output to `tail` / `head` / `grep`.** The project has a PreToolUse hook + capture script that owns cargo output. Run `cargo test ...` bare — no shell plumbing. Piping fights the hook and loses data.
- **Never `pkill -9 -f cargo` to "clear the decks."** That kills Keith's other tmux panes too, not just my processes. Only kill PIDs I personally spawned via `run_in_background`. If I think a test is stuck, ask — don't nuke.

### testing-runner reformats YAML if asked to "fix" (2026-06-01, story 71-24)

### Content-mechanics stories span two repos (story 71-24)
- **A genre-pack mechanics story (`repos: sidequest-content`) also needs the `sidequest-server` test branch pushed.** TEA's failing tests live in `sidequest-server/tests/server/` (they drive the loader + intent-match + seating/dice path); the YAML you author lives in `sidequest-content`. - **A new SWN `melee`-style `combat`+`hp_depletion` bank needs melee WEAPONS with `damage:` specs.** A strike beat with no `damage_override` resolves damage from the actor's inventory weapon (Firefight `shoot` pattern). If the melee weapons (`vibroknife`, `stun_baton`) have no `damage:` field, the bank ablates 0 HP — a decorative bank. Authoring the bank includes giving its weapons damage, or it's half-wired.

A genre-tier mechanics change can be clean while the world still FAILs the structure validator on pre-existing asset gaps — don't conflate the two; check `git diff --name-only` to prove your change didn't cause it.

### The wiring failure class (the dominant bug category in this project)
- **Don't reinvent — wire up what exists.** Before building anything new, grep the codebase. Many systems are fully implemented but not wired into the server or UI. CLAUDE.md rule #3.
- **No half-wired features.** Connect the full pipeline or don't start. If something needs 5 connections, make 5 connections. Shipping 3 and calling it done is the exact failure mode Epic 15 was created to clean up.
- **Wire it, don't just define it.** Adding fields to a struct is not wiring. Adding props to a component is not wiring. Data must actually flow from source → consumer. If no code populates and transmits the field, the feature isn't done.
- **Never rationalize unwired code.** When `/sq-wire-it check` reports FAIL (no non-test consumers), the answer is **wire it**. Do not write paragraphs explaining why this time is different. If I catch myself typing "this is acceptable because" — STOP. That sentence is the bug. The wiring usually takes 5 minutes; the rationalization takes longer.
- **"Wiring means dashboard."** Internal data flow without OTEL visibility is not wired. A missing `WatcherEventBuilder::new(...)` call on a subsystem's dispatch path is a blocking wiring bug, never a "non-blocking improvement." Labeling an OTEL gap as non-blocking in my own audit is the exact self-compounded failure I hit in the 2026-04-09 Map fix.
- **No agent rationalization.** Dev/Architect/TEA/Reviewer are all Claude — they don't get to split blame across personas. "The Architect said it was fine" is not a defense. If a wiring gap is identified in any phase, the only valid response is "fix the code now," never "clarify the spec" or "file a follow-up story."
- **Verify wiring before claiming done.** `grep` for non-test consumers on the merged branch before marking a fix as complete. Don't wait for Keith to ask. A playtest with the GM panel open is the acceptance test.
- **Verify end-to-end before claiming fixed.** Code review is not verification. Trace the full wire path: source → channel → writer task → WebSocket → client handler → DOM. Session 7 had six consecutive "fixed" claims on the turn-lock bug, each found a real issue, none verified the message actually arrived at the browser.
- **Wiring gaps require action, not acknowledgment.** Finding a wiring gap and saying "yep that's a wiring gap" is worthless. Fix it. Log it. Check for siblings (if one compute-then-ignore exists, grep for the pattern). Verify OTEL. Don't move on.
- **Kitchen-sink gate failure is real** (story 16-1): nine agents approved unwired code because every phase rationalized it as "scope boundary" and "wiring comes in subsequent stories." The wiring-check gate exists in `.pennyfarthing/gates/` — every agent must enforce it, not just the reviewer as a backstop.

### No stubs, no fallbacks
- **No stubs. Ever.** When I hit a scope error or type mismatch, solve the actual problem. Don't substitute `Default::default()`, placeholder values, or restructure the design to avoid the issue. CLAUDE.md line 124: "Never say 'the right fix is X' and then do Y." Do X. If `snapshot` isn't in scope, move the code to where it is.
- **No silent fallbacks.** If a path/config/resource isn't where it should be, fail loudly. No `if not exists: try_other_thing`, no `unwrap_or_default()` that papers over missing data, no `Option::None` degradation on required fields.
- **No layered fallback design.** If the LLM is down, the system is down. Keyword intent classification "working" while narration is broken is not meaningful resilience — it's dead code with delusions of usefulness.
- **No reading `~/.pennyfarthing`.** Stay within the project directory tree. Global pennyfarthing config pulls in work-project settings that don't apply to SideQuest.

### Python server test hygiene (2026-04-23 incident — the full-session disaster)
- **30 seconds is the ceiling for the full Python server suite.** Anything above that is a real-service leak, not "the code is slow." The Python side is text munging — prompt composition, JSON, YAML, SQLite — no model inference, no network. Single-digit seconds per test is the normal range; full suite should be tens of seconds, not minutes. If I see a number >30 s, the rule is: **look again**, don't defend it, don't wait to be asked. Diagnosis starts with `ps aux | grep -E "claude|python.*subprocess"` and `lsof /tmp/sidequest-renderer.sock` — catch the leaking real-service subprocess in the act.
- **Diagnostic pytest output must be full.** Don't use `-q` + `tail` when diagnosing slow or failing tests — that swallows per-test timing, the exact captured logs, and the traceback. Use `-v --durations=20` (or `--durations=0` for every test) and read the whole output. `-q | tail` is fine only when you already know the suite is green and fast and you just want the footer.
- **First ask wins.** The very first thing Keith asked about the 6-minute test suite was "are you spawning a claude instance?" — yes, `LocalDM.decompose` inside `_execute_narration_turn` instantiates `ClaudeClient()` by default, which `asyncio.create_subprocess_exec`s a real `claude -p`. Six more explicit pushes followed ("WHY DO WE NEED FASTAPI TRANSPORT", "IT HAS MOCKING", "are you launching webservers?") before I finally acted. If I had trusted the first ask, the session would have been minutes instead of hours.
- **Count the real-service call sites before running anything.** Every slow Python-server test is touching a real service. Full audit list for `SessionHandler._handle_player_action`:
  - `ClaudeClient()` — **instantiated twice per turn**: once by `Orchestrator` (the narrator), once by `LocalDM.decompose` (the decomposer). Both get `self._client_factory()`. If `claude_client_factory` isn't injected, BOTH spawn real `claude -p` subprocesses. Mocking `Orchestrator.run_narration_turn` alone is insufficient — the decomposer runs first.
  - `DaemonClient()` — instantiated inline in `lore_embedding.retrieve_lore_context`, `lore_embedding.embed_pending_fragments`, and `session_handler._run_render`. Default socket path is `/tmp/sidequest-renderer.sock`, default timeout is **180 seconds**. Even a warm daemon blocks for seconds per turn; a dead-socket daemon hangs 3 minutes per call.
  - Real genre pack loading — 15 server test files previously loaded real packs from `sidequest-content/` via `DEFAULT_GENRE_PACK_SEARCH_PATHS`. Pydantic-validates hundreds of YAML files per pack, repeats per test.
- **Guard all three at the conftest layer, not per-test.** `tests/server/conftest.py` now has:
  - Autouse `_mock_daemon_client(monkeypatch)` → replaces `session_handler.DaemonClient` + `lore_embedding.DaemonClient` with `_UnavailableDaemonClient` whose `is_available()` returns False (server's natural fail-fast branch handles that).
  - Autouse `_fixture_pack_search_paths(monkeypatch)` → replaces `DEFAULT_GENRE_PACK_SEARCH_PATHS` with `[tests/fixtures/packs/]` (the frozen fixture pack with symlinks for every real genre slug).
  - Tests MUST inject `claude_client_factory=mock_claude_client_factory(text=...)` into `WebSocketSessionHandler` so BOTH `Orchestrator` AND `LocalDM` get the mock client. This isn't an autouse because constructor-arg injection is cleaner than patching `ClaudeClient` at two import sites.
- **`from X import Y` breaks monkeypatching.** `session_handler.py` does `from sidequest.genre.loader import DEFAULT_GENRE_PACK_SEARCH_PATHS` — that creates a fresh binding in `session_handler`'s namespace. Patching the source module doesn't propagate. Solutions: (a) also patch `sidequest.server.session_handler.DEFAULT_GENRE_PACK_SEARCH_PATHS`, or (b) have session_handler do `import sidequest.genre.loader as _loader_mod` and read `_loader_mod.DEFAULT_...` at lookup time. Same trap exists for any module that imports a class inline (`DaemonClient`).
- **FastAPI is transport, not a test requirement.** `WebSocketSessionHandler.handle_message(msg)` returns `list[object]` — the outbound messages ARE the test target. For 10 of the 15 server tests using `create_app()+TestClient+websocket_connect`, the FastAPI layer added zero coverage and seconds of boot time. The replacement pattern: construct handler with `save_dir`+`genre_pack_search_paths`+`claude_client_factory`, call `attach_room_context(registry, socket_id, asyncio.Queue())`, await `handle_message()`, assert on returned list. Multi-socket tests share a `RoomRegistry`. Tests run in ~0.15s.
- **TestClient survives only for actual HTTP.** `test_rest.py`, `test_games_endpoints.py`, and `test_skeleton_starts.py` genuinely test HTTP endpoints — keep TestClient there. For those, use `app.dependency_overrides` (FastAPI's built-in mocking — https://fastapi.tiangolo.com/advanced/testing-dependencies/) to override heavy dependencies (genre loader, claude factory) so the app boots fast.
- **Fixture genre pack.** `tests/fixtures/packs/test_genre/` is a frozen copy of `mutant_wasteland` (1 world `flickering_reach`, binaries stripped, loads in 128ms). ### Git discipline
- **`cd` persists within a single Bash call.** `cd foo && git -C foo …` resolves to `foo/foo`; the shell resets only *between* calls. Use absolute paths with `git -C` throughout (never mixing a `cd`), or re-anchor with `cd <subdir>` at the start of each chain.
- **Hard ban on `git stash`** — no exceptions, including read-only diagnostics and post-merge "safe" cases. Stashes poof; branches are durable. Set work aside via `git switch -c tmp-<reason>` + commit; to peek upstream use `git show origin/<branch>:<path>`.
- **Never stash during an active merge.** It strips MERGE_HEAD; `stash pop` injects fresh `<<<<<<<` markers on partly-resolved files and the first commit becomes a one-parent commit instead of the two-parent merge GitHub needs to mark the PR mergeable (45-3 incident). Inspect mid-conflict with read-only `git show origin/<branch>:path`.
- **Never stash + checkout develop to check if a failure is "pre-existing."** Wastes time, risks the tree, irrelevant — pre-existing or not, the job is a green suite (Keith, 2026-05-25).
- **Subrepos are independent git repos**, normally on `develop`; a `git commit` inside one lands on *that* subrepo's branch, not the orchestrator feature branch. Per gitflow each subrepo needs a `feat/*` branch. Before dispatching any implementer that touches a subrepo, check `git branch --show-current` and create the matching `feat/<feature>` branch first (2026-05-10: six subagents committed straight to develop).
- **Stacked-branch commits can silently orphan.** Commits that logically belong on parent B but were made on child C don't survive B's squash-merge, then `git rebase --onto develop B C` drops them via patch-id dedup (2026-05-09, 12 commits lost; recovered via `git reflog --all` cherry-pick). Before squash-merging a stacked parent, verify its tip with `git log --stat parent..child -- <files>`.
- **oq-1 and oq-2 are separate orchestrator clones**, each with its OWN subrepos that can drift across branches. Always dispatch subagents / verify with ABSOLUTE paths (`git -C /abs/path`), never bare `cd sidequest-server`. Mass test failures at pack-load are a top symptom of content-clone branch drift — check `git -C .../sidequest-content branch --show-current` FIRST (`main` lacks beneath_sunden/Epic-24 YAML). After `gh pr merge --delete-branch`, gh reverts the tree to the stale local default branch.
- **macOS BSD `sed` silently no-ops `\b`** — use literal patterns + grep-verify. Make a safety branch before any non-trivial rebase.

### Postgres substrate (ADR-115) traps
- **Secondary writes inside an open turn transaction MUST ride that tx, or they self-deadlock under PG (2026-05-26, D8).** `emit_event` opens `with repo.transaction() as tx:` which takes a per-session `FOR UPDATE` row lock on `sessions` (connection A). Any telemetry/side-write reached *inside* that block (e.g. the projection fan-out's visibility-gated `_publish_secret_routed` → `watcher_hub.publish_event` → `PgTelemetrySink.record`) that opens its OWN pooled connection takes a SECOND `session_tx` `FOR UPDATE` on the same locked row → self-deadlock. **Harmless on SQLite** (one in-process connection, no row lock) so it only surfaces under PG on a real MP turn. Fix: thread `tx`/`event_seq` (optional, default `None`) down the call chain so the inner write rides the same connection; lazy-fill / out-of-frame callers keep `None` and safely take their own short `session_tx` (no row lock held). ### Phantom confrontations: the engine CANNOT distinguish a fabricated Other from a legit materialized threat — fix the router (beneath_sunden-6)

The phantom even gets a real backing core (`_seed_combat_hp_depletion_to_npcs` creates one from `cdef.opponent_hp`). So "degenerate dials / no CreatureCore" is the WRONG diagnosis.
- Trap 2 (no clean engine guard): the materialized-threat path (`run_confrontation_dispatch` reads `params["opponent"]|["threat"]` → `instantiate_encounter_from_trigger` materialized_threat branch) serves TWO legit cases structurally identical to the phantom — ship_combat "Raider Frigate" (59-23) and burning_peace's narratively-present-but-unseated watcher (2026-05-31).
