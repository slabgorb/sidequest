### World currency lives in a world `inventory.yaml` `currency:` block — and the world file REPLACES the genre inventory WHOLESALE (2026-06-11, story 103-8)
- Currency is NOT a key on `items.yaml`. `WorldItemsCatalog` (server `genre/models/items.py`) is `extra="forbid"` and only accepts named_items/modifier_items/reliquaries/crimson_remnants/consumable_items/implants. Putting `currency:` there fails the load. (An AC that says "currency as items" is loose phrasing — the real surface is the currency block.)
- The currency pattern is the space_opera one: `worlds/<world>/inventory.yaml` with `currency: {name, denominations[], abbreviation, description, secondary}` (model `CurrencyConfig`, extra=forbid). `denominations` accepts a list or a dict; `secondary` accepts a list/dict/string. aureate_span is the reference.
- **THE TRAP:** `server/dispatch/inventory_resolve.resolve_inventory` does `world.inventory **replaces** the genre one — it is NOT merged.` So a currency-only world inventory.yaml silently wipes the genre's `item_catalog` + `starting_equipment` + `starting_gold`, leaving chargen with no loadouts and no gear. To add a world currency you MUST author the FULL inventory surface (currency + item_catalog + starting_equipment per class + starting_gold), world-flavored. Budget for it — "just add coins" is a full-inventory port.

### World trope `extends:` must be the SLUGIFIED NAME of a genre `tropes.yaml` trope — NOT a seed-trope id (2026-06-11, 103-8)

### Z-Image is an IMAGE LLM, not a reader of prose — author prompts as trained caption labels, LOOK DON'T TELL (2026-06-11, 103-9)
- **It renders nouns and trained style labels, not literary prose.** Keith's rules, learned the hard way on seaboard_of_saints saint portraits:
  - **No analogies / metaphors.** "candle-amber" → a literal CANDLE in every frame. "sea-foam highlight" / "driftwood" / "harbor" → a lighthouse seascape (lighthouse = the most-trained seashore object; the cliche engine defaults to it). Every concrete noun you name gets rendered as that object — only name things you WANT visible.
  - **No inner-thought / mood narration.** "Quiet wonder over dread", "warm and habitable rather than desolate" — invisible to a diffuser, pure waste. LOOK DON'T TELL: describe only what is literally in the frame.
  - **No "in the manner of" / subtle cultural cues.** "WPA post-office mural warmth" is too subtle — it doesn't resolve. 
  - **DO use real trained style labels.** Z-Image was trained on captions like "Italian Renaissance saint portrait", "Byzantine icon", "Sulpician" (French Catholic holy-card devotional style — exactly the saint-card register), "gold-leaf gilt ground", "egg tempera", "devotional" (confirmed works). These produce the centered devotional figure directly. Art-school subject-matter painters bleed their subjects: "Fitz Henry Lane" / "Hudson River School coastal" drag in marine/coastal scenery → drop them from PORTRAIT suffixes.
- **Architecture for separating subject registers:** `render_common.py:92` — a world may declare `portrait_positive_suffix` (portraits only); POIs/illustrations always use the shared `positive_suffix`. So put devotional-icon labels in `portrait_positive_suffix` and leave maritime/lighthouse nouns in `positive_suffix` where POI landscapes legitimately want them. Don't jam portrait framing into the shared suffix.
- **`register:` and `type:` (saint_icon/npc_major) in portrait_manifest are NOT fed to the prompt** by `generate_portrait_images.py:compose_prompt` (it uses name+role+appearance+culture_aesthetic+element_visual only). Devotional framing therefore has to come from `portrait_positive_suffix`, not the per-entry `register`.

### Beating the devotional-icon Christ-face prior + single-portrait iteration (2026-06-11, 103-9)
- **`Byzantine icon` + `saint` = a Christ-Pantocrator face prior so strong it overrides per-entry face description** — every saint came back the same brown-bearded long-haired Christ. Fix: in `portrait_positive_suffix` use `devotional holy card, gold-leaf gilt ground` and DROP `Byzantine icon` and the word `saint`. The gilt gold-ground look survives on the explicit `gold-leaf gilt ground` token alone; the face prior loosens.
- **Negations don't work and backfire in Z-Image.** `no beard` / `clean-shaven` (as a bare adjective) still rendered a full beard (the model sees "beard"). What worked: aggressive POSITIVE smooth-face tokens leading the description — "a long smooth bare chin and hollow clean-shaven cheeks, only a thin dark mustache on the upper lip." That finally produced a beardless Poe.
- **Composition: lead with the frontal figure, cut the backdrop prose.** "...behind him the Acushnet rides at anchor / Ktaadn's tableland stands in cloud" rendered scene-scale (tiny figure in a landscape). Removing it and leading with the figure pulled them back to portraits. Scene-scale subjects (a figure "tall as a water tower", "the Rock at the tideline") need rewriting to a held attribute + frontal figure or they stay landscapes.
- **`--shard k/n` is round-robin over ALPHABETICALLY-SORTED slugs (index % n == k), NOT manifest order.** So `--shard k/31` (k = the portrait's alphabetical index among the world's portrait slugs) renders exactly ONE portrait in ~3 min — the way to iterate a single face cheaply instead of an 87-min full run. Verify which rendered by mtime, not by assuming order.
- Render-cost reality: full 31-portrait `--force` run = ~87 min serial on the MLX daemon (~168s/image). Iterate single faces with `--shard k/31`, do the full run only once the recipe is locked.
