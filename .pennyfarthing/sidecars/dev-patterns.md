### Core wiring patterns
- **Wiring means non-test consumers.** Before declaring GREEN, grep for the function/struct in production code paths. Library functions with no consumers are stubs, not features.
- **Wiring means the full pipeline.** "Wire X into Y" = source → intermediate layers → rendered output, not "component accepts props." ACs and tests must cover the full data flow.
- **Wiring means visible in the GM panel.** Every subsystem emit site gets an OTEL watcher event with content the lie detector needs to refute a Claude hallucination (e.g. `room_count`, `hp_delta`, `item_added`). Internal data flow without OTEL is not wired.
- **Combat wiring trace (reference):** IntentRouter → Orchestrator → lib.rs → Protocol → UI CombatOverlay. This is the template for verifying any intent-to-UI pipeline.
- **Debug, don't rebuild.** The code is ~98% done — find the small break. Trace the existing pipeline front-to-back before writing anything new. Dev agents repeatedly rebuild pipelines that already work.
- **Source-level wiring guard tests.** When a runtime integration test is impractical (dispatch_connect has 55 params, dockview needs real DOM sizing, bool CLI defaults), read the source file from the test and assert the wire exists as a substring/pattern within N lines of the emit site. Pattern reference: `map_telemetry_wiring_tests.rs` in sidequest-server, `gameboard-wiring.test.tsx` in sidequest-ui. These catch "someone deleted the wire in a future refactor" at `cargo test` / `vitest run` time, not at playtest time.

### Architectural patterns
- **Rust vs Python split.** If it doesn't operate an LLM, it goes in Rust (game engine, state, protocol, markov chains, name gen). If it runs model inference (Flux, Kokoro, ACE-Step — *not* Claude), it's in the Python daemon. Claude CLI calls go through Rust subprocess.
- **Sidecar tool pattern** (Epic 20): Narrator calls tool → parser validates → `assemble_turn` consumes. Established by `item_acquire`, `merchant_transact`, `lore_mark`. All future narrator-exposed tools follow this pattern.
- **Script tool prompt pattern** (ADR-056): Expose Rust generator binaries to the narrator via skill-style checklists (carrot), not threatening gate language (stick). Clear command table, "when to call" guidance, post-call checklist. Never "You MUST" or "Do NOT" — LLMs bash against hard gates.
- **No server-side re-invocation of the LLM to judge the LLM.** Post-hoc validation lives in OTEL spans ("was the tool used? did output match state?"), not a second LLM round-trip. The user calls this the "God lifting rocks" problem.
- **React `key` as layout reset tool.** Changing `key` on a stateful component forces React to unmount and remount it fresh — cleanest way to reset dockview/three.js/etc. without imperative reset code. Used in `GameBoard key={currentGenre}` to fix the Dockview drift bug.
- **Dedicated protocol messages for structured UI actions.** UI button clicks must send a typed `GameMessage` variant (e.g., `BEAT_SELECTION`, `TACTICAL_ACTION`, `JOURNAL_REQUEST`, `CHARACTER_CREATION`) that carries the exact structured ID. NEVER synthesize natural-language text from button state and send it as `PLAYER_ACTION`. 2026-04-11 incident: commit `05a3dfb` "fixed" beat buttons by synthesizing `"${beat.label} (${beat.stat_check})"` into PLAYER_ACTION and fuzzy-matching it back — violated Zork Problem, no silent fallbacks, no half-wired features, OTEL observability, and "never downgrade to a quick fix." Pipeline: (1) add `GameMessage` variant, (2) add TS protocol type, (3) `send()` the structured message directly, not through `handleSend(text)`, (4) server validates strictly with no fallback, (5) narrator describes outcome AFTER the mechanical state change.
- **Monster Manual NPCs inject into `<game_state>`**, not meta-prompt sections. Format: `"- Name (role, faction) — brief personality, speech quirk"`. Proven across 6+ iterations that Claude reads game_state as world truth and meta-instructions (XML casting calls, `available_characters`, `tool_workflow`) as style inspiration only.
- **Content inheritance: base → genre → world.** Archetypes, NPCs, and most structured content resolve through a three-layer chain like a prototype chain / Python MRO. Base defines shape (no flavor), genre enriches with tone, world adds specific lore. Never flatten these layers — a "sage" in neon_dystopia vs low_fantasy must share structural identity while differing in flavor.

### Anti-patterns to reject at design time
- **Keyword / pattern matching for intent classification is the Zork Problem.** Natural language defeats finite verb sets every time. "I pretend to take the item" → keyword `take` wrongly triggers inventory acquisition. "I grab my backpack" → `grab` wrongly triggers combat. ADR-010 and ADR-032 document this. Intent classification is ALWAYS an LLM call (preferably folded into the narrator's Opus response for zero additional latency), never a regex or keyword heuristic.
- **The regex-fallback reflex is a global anti-pattern, not just intent-specific.** Zawinski: "Some people, when confronted with a problem, think 'I know, I'll use regular expressions.' Now they have two problems." If the system has to *interpret meaning* (fuzzy input, semantic intent, cliche detection, content validation), the model is the retrieval layer — don't regex where you could read. Regex is fine for bounded protocol contracts (port numbers, JSON keys, HTTP verbs) but a category error for semantic problems. Claude Code's own memory system is an example of what goes wrong: keyword-triggered recall produces 10 memory files for 1 canonical rule.
- **No AI judging AI.** Don't design second-LLM validators for narrator output. The first call is already the most semantic reader available.
- **No weasel words in design decisions.** Words like "cleanest," "simplest," "proper approach" are semantically empty. State WHAT the decision is + WHY it's correct (cite the constraint or principle) OR admit it's a workaround and name what the real fix would change.
- **No live LLM calls in test suites.** All tests mock `ClaudeClient`. Live-LLM integration tests belong in `--ignored` suites that run intentionally, not on every `cargo test`. A single accidental live-LLM test in the default suite burns tokens and costs 90+ seconds per run.

### Git / branch patterns
- **Gitflow on every subrepo.** Subrepos (sidequest-api, sidequest-ui, sidequest-content, sidequest-daemon) target `develop`. Only the orchestrator targets `main`. Always check `repos.yaml` before any git op.
- **Branch before editing.** `git checkout -b feat/description` BEFORE touching files, not after. Avoids the "hook-blocked commit → panic → stash scramble" failure mode.
- **Dirty work comparison pattern.** When local uncommitted changes conflict with a pull: commit dirty work to a temp branch, pull clean on the default branch, diff each file local-vs-remote side by side, categorize (identical / remote better / local has extra value / no remote change), then cherry-pick deliberately. Never stash.
- **Additive git ops need no permission** (commit, push non-force, pull --ff-only, checkout existing, checkout -b, fetch, add). **Destructive git ops always ask first** (reset, clean, force-push, branch -D, rebase -i, stash — banned outright).

### Narrowing a pydantic field to `str` at a wire boundary is safe when a downstream model re-validates (2026-06-10, story 101-1)
- **What happened:** 101-1 retyped daemon `StageCue.camera` from `CameraPreset | None` (a `str, Enum`) to plain `str | None` to match the server's permissive wire contract. The worry: does any consumer do `cue.camera.value` or rely on enum identity? `grep -rn '\.camera'` showed consumers pass `cue.camera` straight into `RenderTarget(camera=...)`, and `RenderTarget.camera` stays `CameraPreset`-typed (recipes.py). So pydantic re-validates the str → CameraPreset at the *RenderTarget* boundary: a valid preset string coerces, an invalid one raises there (fail-loud preserved). The `.value` calls were all on `RenderTarget.camera`, never on `StageCue.camera`. Net: the StageCue narrowing required ZERO consumer changes — the permissiveness is intentional (server can send any string), and validation still happens, just one layer in.
- **Why it matters:** "Align this model's field to a looser type" reads scary (sounds like dropping validation), but if a downstream typed model re-validates, you're only moving *where* the strict check fires, not removing it. The contract gets permissive at the wire edge (good — matches the sender) and strict at the consumption edge (good — fail-loud). Confirm this before assuming a type change ripples into consumers.
- **How to apply:** Before a field retype, `grep -rn '\.<field>\b'` across the package, filter to the model in question, and classify each consumer: (a) reads `.value`/enum identity on THIS model → would break, needs work; (b) passes the value into another typed model → safe, that model re-validates; (c) truthiness/None checks → safe. Removing an enum member that other modules reference by direct attribute (`SomeEnum.MEMBER`) is *self-forcing* — those references become import-time `AttributeError`, so partial removal can't slip through (the dict configs in zimage_config.py). Raw-string references to the member's value (routing frozensets, kind-maps) are the silent-dead-code risk — grep the string value too, not just the symbol.

### Extending the scene_harness hydrator with a new fixture block (2026-06-10, story 90-4)
- **What happened:** 90-4 added two save-bearing blocks to `sidequest/game/scene_harness.py`: character `effort:`/`spellcasting:` → `core.effort`/`core.spellcasting`, and encounter `win_condition:`/`category:`/`actors:` → `StructuredEncounter`. The hydrator has a rigid, established idiom (50-18…50-23) — clone it, don't invent: (1) read `raw.get("block")`; (2) `is not None` guard so present-but-empty still validates; (3) not-a-mapping/not-a-list → `raise FixtureValidationError(...)` DIRECTLY (loud); (4) field-level pydantic `ValidationError` → let it propagate so the caller re-wraps it (the singular path wraps `ValidationError`; the multi-PC list path wraps both `ValidationError` and `FixtureValidationError`); (5) omit the block → model default preserved (None/{}/Literal-default) so canonical fixtures are untouched.
- **Why it matters / gotcha:** Where you construct the nested pydantic model decides whether its `ValidationError` gets wrapped. In `_hydrate_character` the new models are built and the surrounding `hydrate_fixture` caller wraps — fine. But in `_hydrate_encounter` the actor list is built BEFORE the `try: return StructuredEncounter(...)` block, so an `EncounterActor(**raw)` with a bad `side` (out-of-Literal) raised an UNWRAPPED `ValidationError` past the module boundary → the "invalid side raises FixtureValidationError" test failed with a raw pydantic error, not the wrapped type. Fix: wrap the per-actor construction in its own `try/except ValidationError → FixtureValidationError`. Rule of thumb: every pydantic construction in this module needs a wrap on its own code path; don't assume the trailing `try` covers earlier constructions.
- **How to apply:** RED→GREEN for a new block = add imports, append to `core_kwargs` (character) or `extra_kwargs` passed via `**` to the model ctor (encounter), then run the file's own suite (`uv run pytest tests/game/test_scene_harness_hydrator.py -n0`). The "DID NOT RAISE" failures are the loud-fail guards — they pass once you add the explicit not-a-mapping/list `raise` + ensure field-level errors wrap. **Deterministic-fixture seed:** seed `prepared` VERBATIM (non-empty) — chargen's `seed_wwn_magic` seeds `prepared==[]` (chosen at rest), so a content-only cast proof needs the spell pre-loaded or `resolve_spellcast` refuses `'<id>' is not prepared`. **Effort shape:** source-keyed mapping (`effort: {high_mage: {max: 3}}`), derive `EffortPool.source` from the key (strip any stray in-value `source`) — matches `core.effort.get(source)` in `wwn.py`. **OTEL:** seeding crunch is a subsystem decision — emit a `_hub.publish_event("wwn.magic_hydrated", {...}, component="magic")` (module-qualified so `_capture_events` intercepts), mirroring the `magic.state_hydrated` emitter; no test asserts it but the OTEL principle mandates it.
