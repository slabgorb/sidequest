### Core review patterns
- **Wiring verification checklist.** For every story that adds new types/methods: (1) new exports have non-test consumers, (2) new UI components are rendered in the app tree, (3) new hooks are called in production code, (4) new backend subsystems emit OTEL watcher events with state content the GM panel can use as a lie detector.
- **Full pipeline trace.** Trace every hop: emission → channel → writer task → WebSocket → client handler → DOM. For protocol changes, verify serde JSON wire format matches what the client parses. Reference trace: `IntentRouter → Orchestrator → lib.rs → Protocol → UI CombatOverlay` is the template for any intent-to-UI verification.
- **Source-level wiring guard tests are acceptable** when runtime integration is impractical (dispatch_connect has 55 params, dockview needs real DOM sizing). Reference implementations: `map_telemetry_wiring_tests.rs` in sidequest-server, `gameboard-wiring.test.tsx` in sidequest-ui. These catch refactor regressions at `cargo test` / `vitest run` time.
- **No silent fallbacks anywhere.** If the review introduces `unwrap_or_default()`, `Option::None` degradation, `.ok()`, or `if not exists: try_other_thing` patterns, reject it.
- **Wiring gaps require action, not acknowledgment.** Log in sq-wire-it, check for siblings (if one compute-then-ignore exists, grep for the pattern), verify OTEL, then approve.

### Adversarial review patterns
- **Devil's Advocate section is mandatory, not decorative.** Your job is to find the case for rejection. Write it even when the diff looks clean — that's where you discover the subtle issues. The 37-10 UTF-8 panic was found this way (the crash-fix introduced a crash on any response containing multi-byte UTF-8 at byte 200).
- **Rule compliance table.** Walk every applicable rule (silent errors, non_exhaustive, placeholders, tracing, constructors, test quality, unsafe casts, deserialize bypass, public fields, tenant context, workspace deps, dev-deps, validated constructors, fix regressions, unbounded input). Mark each Compliant / Violation / N/A with the entity checked. Inconsistency between sibling entities (e.g., `MutationAction` got `#[non_exhaustive]` but `ExtractionOutcome` didn't) is a violation — call it out.
- **Check regressions specifically.** A diff that fixes bug A while introducing bug B has NOT fixed bug A — it's a net neutral with higher test burden. Blocker for the PR, never a delivery finding.

### Architectural anti-pattern recognition
- **Zork Problem / regex fallback.** When interpretation of meaning is the problem, keyword matching is a category error. If the diff adds keyword heuristics to intent routing, text-matching to dispatch, or regex classification of LLM output — reject. The model that understands semantics is already in the loop; use it. Zawinski's quote applies: "Some people, when confronted with a problem, think 'I know, I'll use regular expressions.' Now they have two problems."
- **Deferral Cascade pattern (sq-wire-it pattern 2).** Every "we'll wire it in the follow-up story" framing is the same pattern that shipped story 16-1 unwired through nine agents. Reject it at first sight.
- **God lifting rocks.** Claude judging Claude's game decisions. Reject any design where a second LLM call validates the first — the first call is already the most semantic reader available.

### Git / branch patterns
- **Gitflow on every subrepo.** Subrepos (sidequest-api, sidequest-ui, sidequest-content, sidequest-daemon) target `develop`. Only the orchestrator targets `main`. Verify PR base branch against `repos.yaml` before approving.
- **Additive git ops are fine** (commit, push non-force, pull --ff-only, checkout existing, checkout -b, fetch, add). **Destructive git ops always require explicit permission** (reset, clean, force-push, branch -D, rebase -i, stash — banned outright). Use `git worktree add` for dirty-tree branch operations.

### Canonical-snapshot lost-update pattern (epic 73)
- **The identity chain that makes a WRITE-tool persistence fix correct:** `room._snapshot` (what `room.save()` writes, `session_room.py:297`) IS `sd.snapshot` because `connect.py:650/690` reassigns `snapshot = room.snapshot` right after `bind_world`; that same reference flows `sd.snapshot → TurnContext.snapshot` (`session_helpers.py:762/1209`) `→ ToolContext.snapshot` (`orchestrator.py:3920`). A WRITE tool that mutates `ctx.snapshot` in place rides the single end-of-turn `room.save()`. Verify every link is a *reference assignment*, not a copy/`model_copy`/re-deserialize — that's the whole ballgame.
- **`ctx.repository.load()` returns a FRESH object, never the canonical.** `PgSaveRepository.load() → load_snapshot()` (`pg/snapshot.py:134-163`) does `SELECT snapshot_json` + `json.loads()` every call. So the old `load()→mutate→save()` tool idiom writes a copy the end-of-turn `room.save()` clobbers = silent lost-update. The fix is mutate-canonical + delete-the-in-tool-save.
- **Known latent gap (filed under epic 73):** ~16 other WRITE tools (`apply_damage`, `apply_status`, `advance_encounter_beat`, `apply_spell_effect`, `apply_world_patch`, `commit_*`, `long_rest`, `tick_tropes`, `update_*`, `veterans_luck`, …) still use the `load()→mutate→save()` fresh-copy idiom. They survive only if `narration_apply` re-applies their effect to canonical — unverified. When reviewing any of these, check whether the effect also lands on the canonical snapshot, or it's the same lost-update class.
- **Opposite-bug check when an in-tool save is *removed*:** confirm an end-of-turn save ALWAYS fires on every production path that can invoke the tool (`websocket_session_handler.py:1171` persistence phase + `:454` disconnect). Removing the save is only safe if the tool is the anomaly and the rest of the turn already relies on the single canonical save.
- **`canonical=True`-style OTEL signals are presence-only, not booleans.** A hardcoded-`True` attribute on the success path that's simply *absent* on the fail-loud path is the right lie-detector shape — but flag docstrings that frame it as distinguishing two values (it can never be `False`).

### Reviewing a pure-data artifact (manifest/JSON), not code (65-7, 2026-06-01)
- **When the diff is a single large data file (e.g. `r2_manifest.json`, 1743 uniform entries), the code-oriented subagents are mostly N/A** — edge-hunter/silent-failure/type-design/simplifier have no code to chew. The review substance shifts to: (1) **schema conformance** exhaustively (rule-checker scripting the file — exact field set, value formats, sort/uniqueness); (2) **source-of-truth correspondence** — *independently* re-derive the truth (re-`list_objects_v2` the live bucket) and diff against the committed file; do NOT accept Dev's self-check when it used the same code that built the artifact (circular); (3) **info-leakage** scan (secrets/tokens/abs-paths) since the artifact is committed+public. I ran the security check myself because the subagent was toggled off — disabled ≠ uncovered for the one domain (security) that applies to data.
- **A "no new tests / wiring-test" gap is non-blocking for a trivial data-population chore when** the artifact is independently verified schema-perfect AND an *audit-green* automated test is impossible due to pre-existing divergence (here `r2_audit` legitimately exits 1 on 1287 content gaps). Demanding the test would bounce a correct, AC-satisfying data commit for a nice-to-have. Record it as a non-blocking Improvement, approve. The "Every Test Suite Needs a Wiring Test" rule governs *code components*; a data file adds no importable/callable component.
- **Separate "the manifest reveals a problem" from "the manifest IS the problem."** rule-checker flagged victoria-orphans, 372 legacy `/images/` paths, source-value — all faithful recordings of R2 reality, not artifact defects. Downgrade-with-rationale (per the no-dismiss rule), route to follow-up findings; don't reject. Stale docstrings in *unchanged* orchestrator files are out-of-scope for a "content repo only" story — editing them would violate the AC.

### Convergent findings are signal; lone high-confidence findings need verification (2026-06-01, story 65-8 review)
- **What happened:** Reviewing the 65-8 manifest-gate. My own pre-subagent adversarial read flagged one gap: the No-Silent-Fallbacks guarantee (POI-bearing world + absent manifest → 500) was unit-tested but never proven **end-to-end**. The reviewer-test-analyzer independently surfaced the *same* gap (#5) plus a missing loud-fail branch (#4). That **convergence** is the strongest signal in a review — two independent passes hitting the same seam.
- **But:** test-analyzer also rated a constant-equality test "tautological / delete it" at HIGH confidence. It wasn't — it mirrors the established 63-8 convention and guards the OTEL **wire-format string** that dashboards depend on. Dismissed with rationale. Meanwhile the rule-checker (sonnet, exhaustive) correctly found the production code clean across all 13 checks and did NOT chase a phantom cross-repo-import "reuse" (which a cheaper simplify pass in the verify phase HAD chased — `scripts/r2_manifest.py` lives in the orchestrator repo, unimportable from the server).
- **How to apply:** (1) Do your own adversarial read BEFORE the subagents return — when a subagent later corroborates your finding, you have high-confidence convergence; when it contradicts, you have a real disagreement to adjudicate with line-level evidence. (2) Treat subagent confidence as a starting point: verify the premise (is the test really vacuous? is the import really possible? is the "inconsistency" real?) before confirming. (3) Grade by **correctness impact**, not the agent's label. (4) Severity discipline holds the line: a pile of real, convergent, MEDIUM test-coverage findings on a correct, wired, rule-clean implementation is APPROVE-with-captured-findings, not REJECT — the `<critical>` wiring-test rule was already met by the route tests; the missing e2e *loud-fail* test is hardening, not the basic wiring proof. Don't inflate Medium to High to justify a reject; capture it as a fast-follow instead.

### "Register-on-import but nobody imports" — green tests over an unreachable feature (2026-06-09, story 86-6 review)
- **What happened:** 86-6 added a `war_rig_crew` table-game kind that self-registers via `register_table_game(...)` at module bottom (the correct pattern). All 83 tests green. But `sidequest/game/table/__init__.py` registers kinds via explicit side-effect imports (`from ...table import poker as _poker`, `... auction`) — and `war_rig` was never added. Production (`instantiate_table_encounter` → `deal_table` → `get_table_game("war_rig_crew")`) would raise `UnknownTableGameError` and 500 the turn. The tests passed only because each test body does `import sidequest.game.table.war_rig` DIRECTLY, executing the registration side-effect that production never triggers.
- **The tell:** a "registration" test that imports the *module under test* directly is NOT a wiring test for side-effect registration — it manufactures the very condition it claims to verify. The true wiring test imports only the *package* (`import sidequest.game.table`) and asserts the kind resolves. CLAUDE.md "Every Test Suite Needs a Wiring Test" + "Verify Wiring, Not Just Existence" exist for exactly this.
- **How to catch it (fast):** for any plugin/registry/decorator-dispatch feature, grep the production registration aggregator (`__init__.py`, a registry module, a `register_*` call site) for the NEW kind/handler name. If the only hits are test files and the self-registering module itself, it's unreachable. `grep -rn "import.*<newmodule>\|<new_kind_string>" sidequest/ | grep -v test`. Convergent here: reviewer-preflight, reviewer-rule-checker (rule 10), and my own grep all hit it independently — strongest possible signal.
- **Severity:** CRITICAL/blocking, not a nit. "It's one import line" is the devil's argument; the aggravating fact is the test suite *actively lies* (green over broken prod). Reject → red rework (TEA writes the package-level wiring test that fails RED, Dev adds the import).

All real, none Critical/High, on a now-correct/wired/rule-clean/green delta. A second rejection for these is the inflation trap — APPROVE and capture them as fast-follow Delivery Findings. The doctrine's job (no green-over-broken-prod) was done.

### Hydrator/pure-data extension: rule-checker is the backstop, verify per-construction-site ValidationError wrapping (2026-06-10, story 90-4)
- **Shape:** 90-4 extended `scene_harness._hydrate_character`/`_hydrate_encounter` with new fixture blocks (WWN effort/spellcasting + hp_depletion win_condition/category/actors). Pure-data pydantic seeding, no async/IO/SQL/new-deps. For this shape only 4 of the 13 lang-review rules even apply (silent-fallback, test-quality, unsafe-deser, input-validation) — the rule-checker (sonnet) is the right tool: it produced an exhaustive 17-rule × 61-instance clean pass and explicitly verified the one subtle correctness point (does any `ValidationError` escape unwrapped past the module boundary?). The load-bearing detail: in this module, field-level pydantic errors are wrapped by the CALLER (`hydrate_fixture`'s singular + multi-PC `except ValidationError`), EXCEPT where a construction precedes the trailing try/except — the encounter `actors` loop, which needs its OWN local wrap. Always trace each new `Model(**raw)` site to the try/except that catches it; a construction that runs before the wrap leaks a raw pydantic error → HTTP 500 instead of the contracted 422. Dev caught this in GREEN (the invalid-side test went unwrapped first); confirm it stayed fixed.
- **Disabled-subagent reality for this project:** `workflow.reviewer_subagents` has edge_hunter, silent_failure_hunter, type_design, security, simplifier OFF; only preflight, test_analyzer, comment_analyzer, rule_checker run. Pre-fill the 5 disabled rows as `Skipped / disabled` in the Subagent Results table — they don't block the completion gate, but the gate DOES require every enabled row at `Received: Yes`. Cover the disabled domains yourself in the assessment (here: type-design + security were trivially clean — pure pydantic models, isinstance guards at every boundary, no secrets/SQL/paths).
- **Preflight that re-runs the full suite:** the preflight subagent's decisive verdict (story-file tests green + ruff + pyright + debug-code grep) lands in its first ~30s; if it then kicks off a redundant full-suite re-run to "confirm pre-existing failures," that's corroboration, not new signal. Read its transcript via `TaskOutput(block:false)` for the captured verdict rather than blocking indefinitely on the re-run — especially when you already verified the pre-existing-failure set independently in an earlier phase (`git diff --name-only develop...HEAD` + `grep -l <module> <failing-files>` is the proof the 7 reds are unrelated).
- **APPROVE on minor-only (doc-accuracy + test-completeness) — don't round-trip.** All confirmed findings here were 3 stale docstrings/comment-over-suggestiveness + 1 untested defensive strip-branch. Real, all minor, on a correct/rule-clean/118-green delta. Capture as fast-follow, APPROVE. The OTEL-wiring-test gap a reviewer would flag (C1) was ALREADY a Dev Delivery Finding — cross-reference existing findings before writing a "new" one.
